**The Interview**

## V1. 첫 구현 — 동작은 하지만 느리다

초기 구현 완료 시점의 구조입니다. 음성을 받아서 AI 응답을 돌려주는 흐름은 완성됐지만, 모든 단계가 **동기 순차**로 실행됐습니다.

<img width="3922" height="376" alt="image" src="https://github.com/user-attachments/assets/cf4c1e8e-086f-4cd0-a80e-5f31f958bc77" />


**문제 1. DB 커넥션이 AI 호출 내내 점유됨**

`@Transactional` 범위 안에 AI 서버 호출이 포함되어 있었습니다. AI 호출은 수 초가 걸리는 네트워크 I/O인데, 그 동안 **DB 커넥션이 그대로 묶여** 있었습니다. 동시 사용자가 늘어날수록 커넥션 풀이 고갈될 구조였습니다.

**문제 2. 같은 빈 내부 @Transactional 무효화**

트랜잭션을 나누려고 같은 서비스 클래스 내부에서 메서드를 분리해 호출했더니 트랜잭션이 적용되지 않았습니다. Spring `@Transactional`은 AOP 프록시 기반이라 같은 빈 내부 호출은 프록시를 우회합니다.

**전체 응답 시간: 8~10초** (AI 팀 TTS 경량화 전 수치)

## V2. Facade 패턴 도입 + 청크 STT

두 가지 변화를 함께 적용했습니다.

### 2-1. Facade로 트랜잭션 경계 분리

별도의 `InterviewFacade` 클래스(Spring이 관리하는 빈)를 두어 **AI 호출 구간을 트랜잭션 밖으로** 꺼냈습니다. **Facade는 트랜잭션 없이 오케스트레이션**만 담당하고, Service의 각 메서드가 짧은 트랜잭션 단위로 호출됩니다.

<img width="230" height="500" alt="image" src="https://github.com/user-attachments/assets/c02722a7-17dc-42d6-a444-598e843e8c63" />


### 2-2. 청크 단위 STT 병렬 처리

기존에는 말이 끝난 후 전체 음성을 업로드하고 STT를 시작했습니다. 이 구조의 문제는 단순히 느린 것이 아니라, **발화 시간이 길어질수록 STT 대기 시간이 선형으로 증가**한다는 점이었습니다. 짧게 답변하면 빠르고 길게 답변하면 느린, 사용자 행동에 따라 응답시간이 들쑥날쑥한 구조였습니다.

말하는 도중 **3초 단위로 청크를 WebSocket으로 전송**해 STT를 선행 처리하도록 바꿨습니다. 발화가 끝나는 시점에 STT가 거의 완료된 상태가 되므로, **발화 시간에 관계없이 응답시간이 균등하게 유지**됩니다.

**청크 분할 기준 선정 과정**

단순히 3초로 정한 게 아니라 두 방식을 **직접 PoC**했습니다.

| 방식 | 문제 |
| --- | --- |
| VAD (무음 감지) | 발화가 길면 청크도 길어져 응답 개선 효과 미미. 사람마다 다른 말 습관에 정확도 불안정 |
| 고정 시간 기준 | 짧을수록 정확도 낮음, 길수록 응답 개선 효과 낮음 |

1초~10초를 직접 테스트한 결과 3~5초 구간에서 정확도와 응답 속도 모두 양호했고, **최대한 짧은 3초**를 선택했습니다.

**비동기 순서 문제 → Redis Sorted Set**

청크 STT를 WebClient로 비동기 호출하면 3번 청크 응답이 2번보다 먼저 올 수 있습니다. 도착 순서대로 합치면 문장이 엉킵니다.

<img width="500" height="700" alt="image" src="https://github.com/user-attachments/assets/102a5851-4597-4575-8206-49d29eff6965" />


ZSet의 score에 청크 인덱스를 사용하면 도착 순서와 무관하게 항상 올바른 순서가 보장됩니다.

**결과: 발화 시간과 응답시간의 선형 비례 관계를 제거. STT 대기 시간 기준 2500ms → 1000ms 이하로 단축, 발화 길이에 무관하게 균등한 응답시간 유지**

## V3. 비동기 체이닝 + 동시성 제어 완성

### 3-1. Race Condition — WebSocket과 HTTP 동시 실행

WebSocket AI 파이프라인(3~5초)이 실행 중일 때 사용자가 종료 버튼을 클릭하면 HTTP 종료 API(0.1초)가 먼저 완료됩니다. 이미 종료된 인터뷰에 꼬리질문이 삽입되는 데이터 무결성 문제가 생겼습니다.

처음에는 DB 상태 컬럼으로 락을 구현했는데 두 가지 문제가 있었습니다.

- `@Transactional` 커밋 전이라 다른 스레드가 변경 전 상태를 읽는 타이밍 문제
- 서버 비정상 종료 시 `IN_PROGRESS`가 DB에 영구히 남아 해당 인터뷰가 영원히 잠기는 문제

위 문제들을 해결하고자 **Redis TTL 락**으로 전환했습니다.

<img width="2500" height="2100" alt="image" src="https://github.com/user-attachments/assets/03455170-6b84-4e31-8f6d-196216e14db2" />


TTL 20초는 정상 파이프라인 최대 완료 시간 약 8초에 여유값을 더한 값입니다. 서버가 비정상 종료되어도 20초 후 **자동 해제**됩니다.

### 3-2. Exactly-Once 트리거 — SETNX

다음 버튼을 누르면 프론트에서 두 이벤트가 **거의 동시**에 발생합니다.

- 마지막 청크 → STT 완료 (빠름)
- 프론트엔드 녹음 종료 처리(`fullRecorder.stop()`) → 전체 음성 blob 생성 → `/next` 전송 (느림)

마지막 STT가 먼저 완료되면 Redis에 `totalChunks` 키가 없어 파이프라인을 트리거할 수 없었습니다. `/next`가 나중에 도착해도 재검사 로직이 없으면 파이프라인이 실행되지 않습니다.

두 경로 모두 트리거 조건을 검사하게 만들되, Redis SETNX(SET if Not eXists, 키가 없을 때만 저장되는 원자적 명령어)로 **중복 실행을 방지**했습니다. 두 스레드가 동시에 시도해도 **정확히 하나만 성공**합니다.

<img width="500" height="1200" alt="image" src="https://github.com/user-attachments/assets/824e4939-9d2c-4e25-b2f4-27ef1103e2c8" />


### 3-3. 독립 AI 호출 병렬화

종료 파이프라인에서 서로 독립적인 AI 호출들이 순차로 실행되어 레이턴시가 합산됐습니다. Reactor의 `Mono.zip()`으로 동시 실행하도록 전환했습니다.

```
순차 실행:  STT 정제 (T1) → 감정 분석 (T2)  →  레이턴시: T1 + T2
병렬 실행:  STT 정제 (T1)                    →  레이턴시: max(T1, T2)
            감정 분석 (T2) ↗
```

키워드 저장, RAG 적재처럼 사용자 응답과 무관한 작업은 `doOnSuccess()` + 별도 `subscribe()`로 메인 체인에서 분리해 **응답을 블로킹하지 않도록** 했습니다.

**BE 단독 개선: 6~8초 → 4~4.5초** (동일 AI 환경 실측)
**전체 개선: 8~10초 → 4~4.5초** (AI 팀 TTS 경량화 포함)

## 기억에 남는 디버깅 — Tomcat WebSocket 1009

React에서만 WebSocket 연결이 Status 1009로 끊기고 세션이 중복 생성되는 현상이 있었습니다. 하지만 바닐라 JS로 동일한 테스트를 하면 정상 동작했습니다.

서버 로그를 두 환경에서 비교했습니다.

| 환경 | 로그 패턴 |
| --- | --- |
| 바닐라 JS `sockjs-client` | 16KB씩 분할 전송, Spring이 메모리에서 조립 후 컨트롤러 전달 |
| React `@stomp/stompjs` | 수백 KB를 단일 프레임으로 전송, Tomcat 레벨에서 즉시 1009 반환 |

원인은 **라이브러리별 전송 방식 차이**였습니다. `@stomp/stompjs`가 보낸 단일 프레임이 Tomcat 기본 버퍼(8KB)를 초과하면 STOMP 핸들러에 도달하기도 전에 Tomcat이 연결을 끊습니다.

Spring WebSocket에는 버퍼 설정 레이어가 두 개 존재합니다.

```
클라이언트
    ↓
Tomcat Container 레벨   ← 여기가 문제였음 (기본 8KB)
    ↓
Spring Transport 레벨   ← 이것만 설정되어 있었음
    ↓
@MessageMapping 컨트롤러
```

`ServletServerContainerFactoryBean`으로 Tomcat Container 레벨의 버퍼의 용량을 늘림으로써 해결할 수 있었습니다.
